# Design: GitHub Actions CI Pipeline

**Date:** 2026-03-15
**Status:** Draft

## Goal

Set up a GitHub Actions CI pipeline that enforces code quality, runs all tests, and builds platform-specific Python wheels (bundling the `ir_dump` C++ binary) for every PR and push to `main`.

## Triggers

```yaml
on:
  push:
    branches: [main]
  pull_request:
```

All four jobs run on every trigger.

## Job Overview

Two independent branches run in parallel:

```
check (ubuntu-latest)           ← ruff + pyright, no Bazel

test (ubuntu-latest)
  ↓
build-binary (matrix: 3 platforms)
  ↓
build-wheel (matrix: 3 platforms)
```

## Job Definitions

### 1. `check`

**Runner:** `ubuntu-latest`
**Needs:** —

Installs Python dev dependencies with `uv` and runs the full suite of static checks:

```
uv sync --group dev
ruff check clickshack/ tests/
ruff format --check clickshack/ tests/
pyright clickshack/
```

No Bazel involvement. Completes in seconds.

### 2. `test`

**Runner:** `ubuntu-latest`
**Needs:** —

Builds `ir_dump` with Bazel, stages it for pytest, then runs all Python tests and the parser smoke suite.

Steps:
1. `actions/checkout`
2. `bazel-contrib/setup-bazel` (with cache config — see Bazel Caching section)
3. Build `ir_dump` and smoke targets:
   ```
   bazel --config=ci build //examples/bootstrap:ir_dump <smoke targets>
   ```
4. Stage binary for pytest:
   ```
   cp bazel-bin/examples/bootstrap/ir_dump clickshack/_bin/ir_dump
   chmod +x clickshack/_bin/ir_dump
   ```
5. `uv sync --group dev`
6. `uv run pytest tests/ -v`
7. `./tools/parser_smoke_suite.sh quick`

### 3. `build-binary`

**Needs:** `test`
**Matrix:**

| `platform` | `runner` |
|---|---|
| `linux-x86_64` | `ubuntu-latest` |
| `linux-aarch64` | `ubuntu-latest` + QEMU |
| `macos-arm64` | `macos-latest` |

Builds `ir_dump` per platform and uploads it as a named artifact for consumption by `build-wheel`.

Steps:
1. `actions/checkout`
2. `docker/setup-qemu-action` (aarch64 only: `if: matrix.platform == 'linux-aarch64'`)
3. `bazel-contrib/setup-bazel` (populates `~/.cache/bazel` on host)
4. **Linux:** Run Bazel inside a manylinux container with cache mounted:
   ```bash
   docker run --rm \
     -v $GITHUB_WORKSPACE:/work \
     -v $HOME/.cache/bazel:$HOME/.cache/bazel \
     -w /work \
     quay.io/pypa/manylinux_2_28_x86_64  # or _aarch64 \
     bazel --config=ci build //examples/bootstrap:ir_dump
   ```
   The volume mount makes host-cached external deps available inside the container, avoiding re-downloads.
5. **macOS:** Native `bazel --config=ci build //examples/bootstrap:ir_dump`
6. `actions/upload-artifact` — uploads `bazel-bin/examples/bootstrap/ir_dump` as `ir_dump-<platform>`

### 4. `build-wheel`

**Needs:** `build-binary`
**Matrix:** same 3 platforms as `build-binary`

Downloads the pre-built binary, stages it, then runs `cibuildwheel` to produce 3 Python wheels per platform (cp311, cp312, cp313).

Steps:
1. `actions/checkout`
2. `docker/setup-qemu-action` (aarch64 only)
3. `actions/download-artifact` — downloads `ir_dump-<platform>` to `bin/ir_dump`, sets executable bit
4. `pypa/cibuildwheel` — the hatch build hook picks up `bin/ir_dump` automatically; no Bazel needed inside the container
5. `actions/upload-artifact` — uploads the 3 `.whl` files produced per run

**Total output:** 9 wheels (3 Python versions × 3 platforms).

## Supporting File Changes

### `.bazelrc` — CI config block

The existing `build --repo_env=PATH=...` line contains a hardcoded local path that breaks on CI. Add a `ci` config block that overrides it:

```ini
build:ci --repo_env=PATH=/usr/local/bin:/usr/bin:/bin
```

All CI workflow jobs pass `--config=ci`. Local builds continue to use the default config (with `cc_no_lld.sh` on macOS).

### `pyproject.toml` — cibuildwheel config

```toml
[tool.cibuildwheel]
build = ["cp311-*", "cp312-*", "cp313-*"]

[tool.cibuildwheel.linux]
manylinux-x86_64-image = "manylinux_2_28"
manylinux-aarch64-image = "manylinux_2_28"
```

The `manylinux_2_28` image matches the image used in `build-binary`, ensuring glibc compatibility between the bundled binary and the wheel platform tag.

## Bazel Caching

`setup-bazel` manages Bazel installation and caches the external repository downloads (abseil, boost, etc.) in `~/.cache/bazel` via GitHub Actions cache. For Linux Docker builds, this directory is volume-mounted into the container so Bazel inside Docker reuses the same cached downloads. Compiled build artifacts are not shared between host and container (they're environment-specific), but those are a smaller cost than re-downloading large external deps.

The `setup-bazel` cache path must be pinned consistently in both the `setup-bazel` step config and the `-v` mount argument.

## Files Changed

| File | Change |
|---|---|
| `.github/workflows/ci.yml` | New — full CI pipeline |
| `.bazelrc` | Add `build:ci` config block |
| `pyproject.toml` | Add `[tool.cibuildwheel]` section |

## Out of Scope

- PyPI publishing / release automation
- Wheel signing or attestation
- Test matrix across multiple OS for the `test` job (Linux only for now)
- Self-hosted arm64 runners (QEMU used instead)
