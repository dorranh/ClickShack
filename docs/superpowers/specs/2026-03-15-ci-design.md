# Design: GitHub Actions CI Pipeline

**Date:** 2026-03-15
**Status:** Draft

## Goal

Set up a GitHub Actions CI pipeline that enforces code quality, runs all tests, and builds platform-specific Python wheels (bundling the `ir_dump` C++ binary) for every PR and push to `main`.

## Triggers

```yaml
on:
  push:
    branches: [main]
  pull_request:
```

All four jobs run on every trigger.

## Job Overview

Two independent branches run in parallel:

```
check (ubuntu-latest)           ← ruff + pyright, no Bazel

test (ubuntu-latest)
  ↓
build-binary (matrix: 3 platforms)
  ↓
build-wheel (matrix: 3 platforms)
```

## Job Definitions

### 1. `check`

**Runner:** `ubuntu-latest`
**Needs:** —

Installs Python dev dependencies with `uv` and runs the full suite of static checks:

```
uv sync --group dev
ruff check clickshack/ tests/
ruff format --check clickshack/ tests/
pyright clickshack/
```

No Bazel involvement. Completes in seconds.

### 2. `test`

**Runner:** `ubuntu-latest`
**Needs:** —

Builds `ir_dump` and all smoke targets with Bazel in a single invocation, stages the binary for pytest, then runs Python tests and executes the pre-built smoke binaries.

Steps:
1. `actions/checkout`
2. `bazel-contrib/setup-bazel` (with cache config — see Bazel Caching section)
3. Build `ir_dump` and all smoke targets in one call:
   ```
   bazel --config=ci build //examples/bootstrap:ir_dump <smoke targets>
   ```
   All targets are built here. The smoke step below only verifies the resulting artifacts.
4. Stage binary for pytest:
   ```
   cp bazel-bin/examples/bootstrap/ir_dump clickshack/_bin/ir_dump
   chmod +x clickshack/_bin/ir_dump
   ```
5. `uv sync --group dev`
6. `uv run pytest tests/ -v`
7. Verify smoke artifacts — bypassing the script's internal Bazel invocation.
   `tools/parser_smoke_suite.sh` currently re-invokes Bazel with its own hardcoded settings (including `CC=cc_no_lld.sh` and `--output_user_root=/tmp/bazel-root`), which would fail on Linux CI. Importantly, the script only **builds** the smoke targets — it does not execute them. Since step 3 already built all smoke targets via `--config=ci`, step 7 only needs to verify that the expected binaries are present in `bazel-bin/`. The script must be updated to accept a `--skip-build` flag that bypasses the Bazel invocation and performs only the artifact-presence check. The CI step calls:
   ```
   ./tools/parser_smoke_suite.sh --skip-build quick
   ```
   This is a required change to `tools/parser_smoke_suite.sh` and must ship in the same PR as the workflow file.

### 3. `build-binary`

**Needs:** `test`
**Matrix:**

| `platform` | `runner` |
|---|---|
| `linux-x86_64` | `ubuntu-latest` |
| `linux-aarch64` | `ubuntu-latest` + QEMU |
| `macos-arm64` | `macos-14` |

`macos-14` is specified explicitly (not `macos-latest`) because the wheel platform tag depends on the build architecture being correct, and `macos-latest` has historically shifted between x86_64 and arm64.

Builds `ir_dump` per platform and uploads it as a named artifact for consumption by `build-wheel`.

Steps:
1. `actions/checkout`
2. `docker/setup-qemu-action` (aarch64 only: `if: matrix.platform == 'linux-aarch64'`)
3. `bazel-contrib/setup-bazel` — populates the Bazel repository cache on the host at a pinned path (e.g. `~/.cache/bazel`). This path must match the volume mount below.
4. **Linux:** Run Bazel inside a manylinux container with the host cache mounted:
   ```bash
   CACHE_DIR=$HOME/.cache/bazel
   docker run --rm \
     -v $GITHUB_WORKSPACE:/work \
     -v $CACHE_DIR:/root/.cache/bazel \
     -w /work \
     quay.io/pypa/manylinux_2_28_x86_64  # or _aarch64 \
     bazel --config=ci build //examples/bootstrap:ir_dump
   ```
   The host-side path (`$HOME/.cache/bazel`) is what `setup-bazel` populates. The container-side path must be `/root/.cache/bazel` because manylinux containers run as root (`$HOME = /root` inside the container). Using `$HOME:$HOME` for both sides would mount to `/home/runner/.cache/bazel` inside the container, which is not where the container's Bazel looks.
5. **macOS:** Native `bazel --config=ci build //examples/bootstrap:ir_dump`
6. `actions/upload-artifact` — uploads `bazel-bin/examples/bootstrap/ir_dump` as `ir_dump-<platform>`

### 4. `build-wheel`

**Needs:** `build-binary`
**Matrix:** same 3 platforms as `build-binary`

Downloads the pre-built binary, stages it, then runs `cibuildwheel` to produce 3 Python wheels per platform (cp311, cp312, cp313).

Steps:
1. `actions/checkout`
2. `docker/setup-qemu-action` (aarch64 only)
3. Create staging directory and download artifact:
   ```
   mkdir -p bin
   actions/download-artifact  # with path: bin
   chmod +x bin/ir_dump
   ```
   `bin/` is gitignored and will not exist after checkout; it must be created before `download-artifact` runs. The `download-artifact` step must set `path: bin` — without this the action defaults to creating a subdirectory named after the artifact (e.g. `ir_dump-linux-x86_64/ir_dump`), which is not where the hatch build hook looks.
4. `pypa/cibuildwheel` — the hatch build hook picks up `bin/ir_dump` automatically; no Bazel needed inside the container
5. `actions/upload-artifact` — uploads the 3 `.whl` files produced per run

**Total output:** 9 wheels (3 Python versions × 3 platforms).

## Supporting File Changes

### `.bazelrc` — CI config block

The existing `build --repo_env=PATH=...` line contains a hardcoded local path that breaks on CI. Additionally, `build --repo_env=CC=cc_no_lld.sh` and `CXX=cc_no_lld.sh` are unconditional — `cc_no_lld.sh` is a macOS-specific wrapper that does not exist on Linux runners.

Add a `ci` config block that overrides all three:

```ini
build:ci --repo_env=PATH=/usr/local/bin:/usr/bin:/bin
build:ci --repo_env=CC=cc
build:ci --repo_env=CXX=c++
```

All CI workflow jobs pass `--config=ci`. Local builds continue to use the default config (with `cc_no_lld.sh` on macOS).

### `pyproject.toml` — cibuildwheel config

```toml
[tool.cibuildwheel]
build = ["cp311-*", "cp312-*", "cp313-*"]

[tool.cibuildwheel.linux]
manylinux-x86_64-image = "manylinux_2_28"
manylinux-aarch64-image = "manylinux_2_28"
```

The `manylinux_2_28` image matches the image used in `build-binary`, ensuring glibc compatibility between the bundled binary and the wheel platform tag.

### `tools/parser_smoke_suite.sh` — `--skip-build` flag

The script currently re-invokes Bazel with hardcoded settings incompatible with Linux CI. A `--skip-build` flag must be added that bypasses the Bazel build step and directly executes already-built binaries from `bazel-bin/`. The CI `test` job builds all required targets in step 3 and then calls the script with `--skip-build`.

## Bazel Caching

`setup-bazel` manages Bazel installation and caches the external repository downloads (abseil, boost, etc.) on the host via GitHub Actions cache. For Linux Docker builds, this directory is volume-mounted into the container so Bazel inside Docker reuses the same cached downloads. Compiled build artifacts are not shared between host and container (they're environment-specific), but external dep downloads are the dominant cost.

The `setup-bazel` cache path must be pinned explicitly in both the `setup-bazel` step config and the `-v` mount argument. The container-side mount target must be `/root/.cache/bazel` (not `$HOME/.cache/bazel`) because manylinux containers run as root.

## Files Changed

| File | Change |
|---|---|
| `.github/workflows/ci.yml` | New — full CI pipeline |
| `.bazelrc` | Add `build:ci` config block (PATH + CC + CXX) |
| `pyproject.toml` | Add `[tool.cibuildwheel]` section |
| `tools/parser_smoke_suite.sh` | Add `--skip-build` flag for CI use |

## Out of Scope

- PyPI publishing / release automation
- Wheel signing or attestation
- Test matrix across multiple OS for the `test` job (Linux only for now)
- Self-hosted arm64 runners (QEMU used instead)
- Runtime validation of manylinux `ir_dump` binaries — `build-binary` verifies the Bazel build succeeded; it does not execute the binary inside the container or run smoke tests against it. The manylinux binary goes directly to `build-wheel` without runtime verification.
