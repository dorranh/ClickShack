# Phase 02.1 Research: Select Language Feature Completeness (INSERTED)

## Objective
Plan and execute a parser-only completeness sweep for the SELECT dialect gaps listed in `docs/select_dialect_missing_features.md`, while preserving deterministic workload validation and explicit exclusions.

## What We Need To Know Before Planning

1. The phase should be run as a broad gap-closure sweep (per `02.1-CONTEXT.md`), not a narrow top-N patch.
2. The current parser already accepts some syntax but drops AST fidelity for critical fields (ORDER BY modifiers, WITH RECURSIVE metadata, SELECT ALL intent, WINDOW internals).
3. Validation currently depends on canonical summary outputs (`parser_workload_summary`) and fixture metadata (`manifest.json`), so AST changes must be mirrored in summary output and fixture expectations.
4. The largest architectural decision is whether to keep some fields as parser-acceptance-only or persist them in AST now; phase context requires parity-evidence, so retention should be preferred for listed gaps.
5. Exclusions must remain explicit and deterministic (`non_sql`, `non_select`, `mssql`, `query_parameter`) unless roadmap scope changes.

## Baseline Findings From Current Code

- `ParserSelectRichQuery.cpp` parses `WITH RECURSIVE` and `SELECT ALL`, but `ASTSelectRichQuery` has no fields preserving those flags.
- `SelectClausesLiteParsers.cpp` accepts `NULLS FIRST|LAST`, `COLLATE`, `WITH FILL`, and `INTERPOLATE` but intentionally ignores payload in AST.
- `SelectClausesLiteParsers.cpp` currently parses LIMIT/OFFSET/FETCH via numeric token extraction (`parseNumberToken`), not general expressions.
- `TableSourcesLiteParsers.cpp` supports `GLOBAL` joins and common join types but not `LOCAL` modifier or `PASTE JOIN`; parenthesized table atoms currently assume nested SELECT/WITH subqueries.
- `ExpressionOpsLiteParsers.cpp` supports searched CASE only (`CASE WHEN ...`); simple CASE (`CASE expr WHEN ...`) and window application (`... OVER (...)`) are absent.
- `ASTLiteral` has only `Number` and `String`, so NULL is represented as string payload.

## Implementation Workstreams and Sequencing

### Workstream 1: AST Contract Expansion (do first)
Purpose: define durable storage for features currently parser-accepted but dropped.

Targets:
- Add SELECT-head fidelity fields to `ASTSelectRichQuery` (`select_all_explicit`, `with_recursive`).
- Extend `ASTOrderByElementLite` to retain:
  - nulls placement (`NONE|FIRST|LAST`)
  - optional collation identifier
  - WITH FILL payload (`from`, `to`, `step`, `staleness`)
  - optional INTERPOLATE expression list
- Replace `ASTWindowDefinitionLite::body` raw string with structured members (partition/order/frame subnodes or minimally typed components) if feasible in phase scope; otherwise add explicit TODO boundary and summary-level evidence.
- Extend `ASTJoinLite` to encode `LOCAL` vs `GLOBAL` join locality and `PASTE` join type.
- Add NULL kind to `ASTLiteral::Kind`.
- Evaluate LIMIT AST shape:
  - `ASTLimitLite`/`ASTLimitByLite` currently store strings; move to `IAST *` expression fields (or add parallel expression fields) if expression-flexibility gap is in scope for full completion.

Why first: downstream parser work becomes mechanical once AST destinations exist.

### Workstream 2: SELECT Head + WITH Surface
Targets:
- `ParserSelectRichQuery.cpp`
  - parse and persist `DISTINCT ON (expr_list)`
  - parse and persist `TOP n [WITH TIES]`
  - persist explicit `SELECT ALL`
  - persist `WITH RECURSIVE`
  - improve WITH alias retention (`WITH expr AS alias`, bare alias) using explicit AST wrapper rather than alias-discard behavior

Sequencing note:
- Implement `DISTINCT ON` and `TOP` in mutually exclusive resolution order compatible with ClickHouse precedence.

### Workstream 3: FROM/JOIN Surface
Targets:
- `TableSourcesLiteParsers.cpp`
  - add `LOCAL` join modifier support (parallel to existing GLOBAL path)
  - add `PASTE JOIN` operator mapping
  - allow parenthesized table expressions that are not subqueries when dialect permits
- `ASTJoinLite.h` and consumers (`select_rich_smoke.cc`, `parser_workload_summary.cc`) to report new join locality/type dimensions.

### Workstream 4: Expression + Window Support
Targets:
- `ExpressionOpsLiteParsers.cpp`
  - add simple CASE (`CASE <expr> WHEN ...`) lowering to deterministic AST shape
  - add `OVER (...)` and `OVER window_name` parsing for function/window expressions
  - keep precedence and boundary behavior deterministic (avoid consuming clause keywords)
- `SelectClausesLiteParsers.cpp`
  - parse `WINDOW` definitions into structured AST (or strongly typed intermediate nodes)

### Workstream 5: ORDER BY, GROUP BY, LIMIT/OFFSET Fidelity
Targets:
- `SelectClausesLiteParsers.cpp`
  - retain `NULLS FIRST|LAST`, `COLLATE`, `WITH FILL`, `INTERPOLATE` in AST nodes
  - add `ORDER BY ALL`
  - add `GROUPING SETS (...)`
  - broaden LIMIT/OFFSET/FETCH value parsing from numeric-token-only to expression-capable parse path
- `ASTOrderByElementLite.h`, `ASTLimitLite.h`, `ASTLimitByLite.h` updates as needed.

### Workstream 6: Validation + Fixture + Docs Closure (final wave)
Targets:
- Add supported fixtures for each newly implemented family.
- Reclassify exclusions if scope has changed; otherwise keep existing exclusion tags intact.
- Update expected canonical summaries and parser summary emitters.
- Update docs for exclusion boundaries and completion evidence.

## Test/Validation Strategy (Aligned to Existing Workload Suite)

1. Keep `manifest.json` as the contract source for fixture scope and expected canonical summary.
2. For each gap family, add at least:
- one success fixture demonstrating parse + AST retention,
- one edge fixture covering ambiguity/precedence,
- exclusion fixture only when intentionally still out of scope.
3. Extend `parser_workload_summary.cc` so new AST fields are emitted in deterministic canonical order.
4. Validate with existing suite commands in this order:
- `tools/parser_workload_suite.sh audit --manifest testdata/parser_workload/manifest.json`
- `tools/parser_workload_suite.sh supported --manifest ...`
- `tools/parser_workload_suite.sh excluded --manifest ...`
- `tools/parser_workload_suite.sh repeat --manifest ...`
- `tools/parser_workload_suite.sh full --manifest ... --source ... --signoff ...`
5. Preserve baseline build/smoke gates:
- `just smoke-suite`
- `bazel build //ported_clickhouse:parser_lib //examples/bootstrap:parser_workload_summary`

## Validation Architecture

1. Contract Layer (manifest + docs)
- `manifest.json` defines fixture identity, class (`success|excluded`), expected canonical summary/failure class.
- `audit` and `audit-docs` remain mandatory before parser execution to prevent fixture/doc drift.

2. Parse Behavior Layer (supported/excluded)
- `supported` enforces parse success + full token consumption + exact canonical summary match.
- `excluded` enforces deterministic failure class for declared out-of-scope syntax.

3. Determinism Layer (repeat)
- `repeat` executes supported fixtures multiple times and compares exact summary output.
- Any new AST field must be stable under repeated parse of identical input.

4. System Regression Layer (build/smoke)
- Run `just smoke-suite` and parser library build to ensure no regressions in existing smoke entrypoints.
- Keep `parser_workload_summary` as the single canonical emitter; avoid duplicate summary logic across binaries.

5. Evidence Layer (phase artifacts)
- Update scope/reconciliation/validation docs and signoff artifacts to reflect new supported families and remaining exclusions.
- Evidence must show both acceptance and AST-retention fidelity for touched features.

## Risks and Mitigation

1. Risk: AST surface expansion breaks canonical summary compatibility.
- Mitigation: add fields behind deterministic default values and update fixture expectations in the same commit wave.

2. Risk: Grammar ambiguity around TOP, DISTINCT ON, and existing LIMIT/FETCH forms.
- Mitigation: add precedence-focused fixtures first; implement parser order mirroring upstream parse shape.

3. Risk: Over-scoping into non-SELECT or semantic-analysis behavior.
- Mitigation: keep phase checklist tied to `docs/select_dialect_missing_features.md`; reject analyzer/runtime tasks during planning.

4. Risk: LIMIT/OFFSET expression upgrade causes broad AST churn.
- Mitigation: decide early whether to add parallel expression fields (non-breaking) vs replacing string fields (breaking); choose one and lock in plan.

5. Risk: Window structuring effort explodes due to frame grammar complexity.
- Mitigation: plan incremental structure with bounded fidelity target and explicit deferral notes only if unavoidable.

6. Risk: Hidden exclusion regressions (query parameters, MSSQL, non-SELECT) while widening parser acceptance.
- Mitigation: retain exclusion fixtures and failure-class assertions in every full suite run.

## Recommended Planning Defaults

- Plan in 3 execution waves:
  1. AST contract + parser hooks for SELECT/JOIN/ORDER/GROUP gaps,
  2. expression/window/limit fidelity,
  3. fixture expansion + deterministic validation + docs/signoff.
- Require each wave to end with `supported + excluded + repeat` before proceeding.
- Treat remaining unimplemented items as explicit exclusions with documented rationale; never leave silent parser-accept/drop behavior.
