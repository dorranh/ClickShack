# Phase 02.1: Select language feature completeness - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Close urgent SELECT-language completeness gaps in the parser-only bridge, using `docs/select_dialect_missing_features.md` as the authoritative scope list for missing/incomplete features relative to upstream ClickHouse parser behavior. This phase clarifies parser coverage and AST fidelity within SELECT scope only; no analyzer/runtime semantics and no non-SELECT scope expansion.

</domain>

<decisions>
## Implementation Decisions

### Coverage Scope Source
- `docs/select_dialect_missing_features.md` is the canonical scope anchor for this phase.
- Planning should treat the documented gap set as the phase input surface rather than re-discovering candidate features.

### Priority and Breadth
- Execute a broad sweep over the documented SELECT gaps in this urgent insertion phase rather than limiting to a top-5 subset.
- Sequence work to preserve parser build/test stability while maximizing feature completeness progress across modifiers, joins, analytic/window expressions, ORDER/GROUP BY fidelity, and expression/limit fidelity.

### Definition of Done
- Completion target is a credible parser parity claim for SELECT language surface (within explicit project boundaries), not just incremental fixture pass.
- Verification should include both parse acceptance behavior and AST retention/fidelity evidence for touched features.

### Unsupported Feature Policy
- Any remaining non-implemented items after this phase must stay explicitly classified as exclusions (not silent/soft unsupported).
- Exclusions must be deterministic in fixture behavior and reflected in docs/signoff artifacts.

### Claude's Discretion
- Exact plan slicing/waves and implementation order inside the broad sweep.
- Specific test fixture granularity and grouping strategy, as long as parity-claim evidence remains auditable.

</decisions>
